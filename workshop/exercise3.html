Location: Sarah's desk, ACME Financial HQ. Time: 07:23 AM. The caffeine is finally starting to kick in, but the tension from the 2:00 AM page is still heavy in the air.

Morgan: (Leaning over Sarah’s shoulder, eyes fixed on the web console) Sarah, look, the egress logs are quiet now thanks to that Network Policy you slammed down, but we’re flying blind in the SOC on what actually happened inside that container. If we restart that pod, the threat actors toolkit is gone forever. I need a forensic checkpoint.

Sarah: (Rubbing her eyes) I hear you, Morgan. We’ve got the PV snapshot done, so the disk is "frozen," but you want the full runtime state, right? The RAM, the open file descriptors, the works?

Morgan: Exactly. In the old days, I’d just dump the physical memory of the server. But here? This "container" is just a slice of a kernel. How do we pull the memory out of a running pod without killing the process?

Sarah: I think it’s called a CRIU checkpoint. It stands for Checkpoint/Restore In Userspace. Theoretically, it freezes the container and serializes its entire state into a set of image files.

Morgan: (Nodding) That’s the ticket. Can you run it? I want to see if they left a reverse shell sitting in the heap.

Sarah: To be honest... I’ve never actually done this before. OpenShift has some specific security gatekeeping for this because, well, dumping memory is a sensitive operation. Let me pull up the docs.

3.1 Checkpointing the breached pod

Container checkpointing was introduced as an alpha feature in Kubernetes v1.25 and graduated to beta in Kubernetes v1.30 and is a Generally Available feature in OpenShift 4.17 onwards. This functionality allows running containers to be transparently checkpointed to persistent storage and later restored to resume execution, or migrated across nodes and clusters.

The content of container checkpoints can be further analyzed with the checkpointctl tool. This allows to perform forensic analysis in case of security incidents (e.g., suspected compromise, data exfiltration) or application failures by inspecting the saved process memory, open files, sockets, and execution context captured in the checkpoint.

Sarah: OK here we go, I need to use the kubelet API directly. I have to run a curl command against the local kubelet API endpoint on the specific worker node where the payment-gateway is sitting.

Morgan: A curl command to dump memory? That’s... surprisingly old school for such a modern platform!

Sarah: (Laughing weakly) Welcome to Kubernetes. The command looks like this: curl -X POST "https://localhost:10250/checkpoint/payment-gateway-x/payment-gateway/httpd". I’ll need to do this from a debug shell on the node itself. I think there is a helper script on a Red Hat support article I can use to make it easy.

Morgan: Will the container stay alive after the checkpoint?

Sarah: The docs say it depends on the flags, but for a "forensic" checkpoint, the goal is to keep it running in its "frozen" state or let it continue. Since we already have the egress blocked, I’ll let it keep running so we don't tip off any watchdog scripts the attacker might have left.

Morgan: Sounds good, do it.

Sarah: Initiating the node-level debug shell now. Hang on... and... started.

Initiate a checkpoint by copying and running the following script in your OpenShift Web Terminal:

NODE_NAME=$(oc get pods -l app=payment-gateway -o jsonpath='{.items[0].spec.nodeName}')
CURRENT_NS=$(oc project --short)

echo "==> Triggering fresh CRIU checkpoint on node: $NODE_NAME..."

API_RESPONSE=$(curl --silent --insecure \
  --header "Authorization: Bearer $(oc whoami --show-token)" \
  --request POST "$(oc whoami --show-server)/api/v1/nodes/${NODE_NAME}/proxy/checkpoint/${CURRENT_NS}/payment-gateway/httpd")

TARBALL_PATH=$(echo "$API_RESPONSE" | jq -r '.items[0] // empty')

if [ -z "$TARBALL_PATH" ] || [ "$TARBALL_PATH" = "null" ]; then
  echo "[!] ERROR: Checkpoint failed. API response was:"
  echo "$API_RESPONSE"
fi

CHECKPOINT_FILE=${TARBALL_PATH##*/}

echo "==> Checkpoint created successfully: $CHECKPOINT_FILE"
echo "==> Streaming memory dump into Web Terminal..."

oc debug node/"$NODE_NAME" --to-namespace="$CURRENT_NS" -- \
  chroot /host /bin/sh -c "cat $TARBALL_PATH" > "/tmp/$CHECKPOINT_FILE"

echo "==> Success! Verifying saved forensic artifact in /tmp/:"
ls -lah "/tmp/$CHECKPOINT_FILE"

After a short wait you should see that a checkpoint has been created.

3.2 Exploring the checkpoint

Sarah: Alright Morgan, we successfully streamed the container's memory checkpoint down to the terminal. Now we need to figure out exactly what the attacker was running.

Morgan: Perfect! We can use the checkpointctl tool to take a look at it.

Run the script below in your web terminal to make the checkpointctl binary available.

echo "==> Fetching precompiled checkpointctl toolkit..."
curl --silent --fail --location \
  'https://raw.githubusercontent.com/jmhbnz/checkpointctl/main/bin/checkpointctl' \
  --output /tmp/checkpointctl

chmod +x /tmp/checkpointctl
mkdir -p $HOME/bin
mv /tmp/checkpointctl $HOME/bin/checkpointctl
export PATH=$HOME/bin:$PATH

echo "==> Toolkit loaded successfully!"
checkpointctl

echo "--------------------------------------------------------"
echo "==> To inspect your captured memory dump, run:"
echo "checkpointctl inspect /tmp/\$CHECKPOINT_FILE"

Once checkpointctl is installed in your web terminal run the command below to inspect the process tree of your checkpoint(s).

checkpointctl inspect --ps-tree-env /tmp/checkpoint-payment-gateway_payment-gateway*

You should see a summary of the process tree that was running inside the container at the point in time the checkpoint was taken, an example is below. Of interest from a forensics perspective is the process with id 235 which is running a bash script exfil.sh.

Sarah: This is cool, I've never played with checkpoints like this before. We have some good metadata there. It shows the container runtime was CRI-O, gives us the exact timestamp of the snapshot, and lists the root process.

Morgan: Check out the process tree section—there's a PID running a script called exfil.sh. That's our smoking gun. Let's dig into the memory pages next to extract the environment variables and network sockets they left behind. Leave that to me and the Infosec team, we need to go through this with a fine tooth comb and document everything.

3.3 Forensics resources

From a platform administration perspective our goal here is become confident and practice our skills taking forensics checkpoints.

If you want to dive deeper into the world of volatile memory forensics and runtime snapshots, there are fantastic open-source ecosystems to explore. To learn more about how container checkpoints are structured under the hood, check out the official CRIU (Checkpoint/Restore In Userspace) project documentation at criu.org, which details how Linux process states are frozen and resurrected.

For practical hands-on security analysis, look into the Volatility Framework, the industry-standard memory forensics platform, as well as the CNCF Kubernetes Security Specialist (CKS) learning tracks, which frequently cover container runtime security, node isolation, and live incident response strategies.

Feel free to take some time now exploring those resources. When you are ready let's move on to Exercise 4!
